## Key Decisions Already Made (don't re-litigate)

- Milestone ORDER was deliberate: error surfacing (M1) → detection wiring
  (M2) → the dangerous switch making conflicts triggerable (M3) last.
- Conflict key is an honest tuple `(u8 page_type, PageId, u16)`.
- `log_label_write` must NOT record write sets (idempotent label dict).
- `StorageAccess::commit_tx` uses `Result<(), String>` (keeps graft-query
  decoupled from graft-core).
- On commit failure the executor must NOT call `abort_tx()` —
  `tx_mgr.commit()` already aborted internally and shipped a replication
  Abort.
- Cluster commit paths drive ALL shards even when one fails, return first
  error.
- M3: transient per-call tx context on ShardCluster (fields kept,
  semantics changed) instead of threading tx_id through graft-query's
  StorageAccess trait — reuses executor plumbing unchanged. Per-connection
  tx state lives with the connection (server's `active_tx`), passed into
  the tx_id-parameterized cluster methods.
- Server's "transaction already active" guard on BEGIN is per-connection
  wire-protocol policy — kept.
- `Shard.tx_mgr` stays `pub(crate)` (event_loop.rs tests inject write sets
  to test commit-result plumbing in isolation).

## Pitfalls to Avoid

- All of tasks/lessons.md still applies. Highlights: graft-server has no
  lib target (integration tests in tests/, no src unit tests exposed);
  check clippy failures against main via git stash before assuming they're
  yours; any new persisted file needs a `.wal`/`.meta` extension or
  `is_data_file()` in crates/graft-io/src/uring.rs grows a case; RefCell
  function-scoped borrows need explicit drop() before &mut self helper
  calls; broad replace_all edits FIRST, targeted exceptions second.
- NEW (M3): conflict tests that assert post-conflict VALUES must order
  writes so the WINNER writes last — property records are overwritten in
  place with no MVCC versioning (pre-existing gap, out of scope), so the
  loser's physical bytes would otherwise be what a read returns even
  though its commit correctly errored. Commit Ok/Err assertions are
  ordering-independent.
- Replica clusters seed `next_tx_id` at `REPLICA_TX_ID_BASE = 1 << 63`
  (disjoint tx-id ranges by construction) — M3 preserved this; don't break
  it in any M4 rework of commit paths.
- Pre-existing adjacent gap, NOT part of this task: property records carry
  no tx_min/tx_max (see above). Flag, don't fix.
